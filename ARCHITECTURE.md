## Entry Points

The primary main scene is [`scenes/init/init.tscn`](/scenes/init). Under normal circumstances, this is where the game initializes itself from.

[`scenes/documentation/init.tscn`](/scenes/documentation) is ran by contributors when needing to develop a feature in isolation inside a scene. We follow the concept of [Gyms, Zoos, and Museums](https://youtu.be/5PJRCz0t7yY) to test things in the game, to define how they should behave, and to document them both for ourselves and outsiders. This concept is used both within the [AAA games industry](https://minecraft.wiki/w/Debug_mode) and in [indie games](https://getasnack.fandom.com/wiki/Extras_(SNACKCORE)).

No other official entry points exist in this point in time.

### `examples` ⚙️🛠️

Examples provided by addons.

### `globals` 🛠️👾

Autoloaded nodes. These are created once but exist in every scene.

### `modules` 🛠️👾

Self-contained reusable feature or aspect of the game. The first level of subfolders here are named after the aspect of the game or feature that they encompass.

This helps to make it so that not only is it easier for others to read what you've made but also easier to update as contributors no longer need to deal with everything sloppily placed under a single folder; in other words, we can know which file relates to what file.

To make sure everyone is on the same page when creating modules follow these rules.

- **Do not write code using AI**. It has encouraged the creation of sloppy hard to read code in the past, and shall not be used ever in the future.
- Use `snake_case` for all of your folders and file names.
- Follow the folder structure:

```
modules ~
└─ [module_name]
   ├─ data      :: Do try limiting it to just providing default data.
   ├─ examples  ::
   ├─ nodes     ::
   ├─ resources ::
   ├─ scenes    ::
   ├─ screens   ::
   ├─ scripts   ::
   └─ views     ::
```

By practicing a bit of discipline we can simplify long-term maintainence costs.

**note`(1)`**: This folder was formerly named `libraries`. It was renamed to `modules` to further emphasize the self-contained but reusable aspect in the hopes of reducing the minimal but still substantial technical jargon.

### `nodes` 🛠️👾

Code that represents some form of in-game object. Note that nodes do not necessarily have to be perceived (seen or heard) by the player, just that it exists within the world that they're playing in or more precisely within the scene tree.

### `resources` 🛠️👾

Files here are used to define your own custom data container types, the shape of a custom data container type, and the operations one can perform with them.

[`Resource`s](https://docs.godotengine.org/en/stable/tutorials/scripting/resources.html) are more lightweight than [`Node`s](https://docs.godotengine.org/en/stable/classes/class_node.html#class-node) but still heavier than [a plain old script extending nothing in particular](https://docs.godotengine.org/en/stable/classes/class_refcounted.html). These have special integration with the Godot Editor that makes it easier to edit the data contained within them.

### `screens` 🛠️👾

## Cross-Cutting Concerns

This section contains details that are found everywhere within the codebase but are not specific to any set of files or folders.

### Versioning

`YYYY-MM-DD-HH-MM_UUID`, no fuss or misguided attempts at being clever.

The only time a desktop and/or mobile application such as games needs to have semantic versioning is gaining the support of a manager with a non-technical background.

Semantic versioning — v`MAJOR.MINOR.PATCH` notation — makes sense for software that exposes a publicly available Application Programming Interface (API) where end users of the API have to know the distinction between a **major** update with a large set of changes that can break their existing code, a **minor** update that adds features or removes a deprecated feature, or a **patch** update for bug fixes.

Under no circumstance should any contributor to the repository *sabotage* it by implementing semantic versioning to the game itself. You *can* implement semantic versioning only in the `modules` folder where it makes sense.

For players, what they're looking for is how recent is any given update. Simply versioning it according to the timestamp + a unique ID is enough.

### Testing

Automated testing is done with the [`gdUnit4`](https://github.com/godot-gdunit-labs/gdUnit4) testing framework complemented by play testing.

When writing tests, one pitfall you should avoid is testing for implementation details when you should be testing for expected behaviors. Implementation details refer to things callers of your code will never know or see about and shouldn't care about; in other words, test for the result not how it's done and see if it lines up with how it should behave.

**It is highly discouraged to create your own testing solutions without consulting the [`gdUnit4` documentation](https://godot-gdunit-labs.github.io/gdUnit4/latest/advanced_testing/index/) first.** You'll be creating additional code to test for your own ad-hoc mini testing framework, and it won't have the benefit of being as widely used whilst still holding you liable for when it breaks. **I repeat, consult the [`gdUnit4` documentation](https://godot-gdunit-labs.github.io/gdUnit4/latest/advanced_testing/index/) first.**

Avoid flaky tests (that is — tests that inconsistently pass or fail despite being given the exact same inputs). Flaky tests often involve I/O (disk read/write, network requests, etc.). You may isolate and test the parts that are pure (for every given input returns the same output, and does not alter any data outside its own scope)

### Performance Metrics

[The great Donald Knuth has said](https://docs.godotengine.org/en/stable/tutorials/performance/general_optimization.html#principles)

> Programmers waste enormous amounts of time thinking about, or worrying about, the speed of noncritical parts of their programs, and these attempts at efficiency actually have a strong negative impact when debugging and maintenance are considered. We should forget about small efficiencies, say about 97% of the time: **premature optimization is the root of all evil.** Yet we should not pass up our opportunities in that critical 3%.

Do not overthink about optimization. Optimization is important but concentrate your efforts on aspects that actually have an impact on the game.

Our time is limited, we can accept a slightly worse loading time if it means that we can get the game run at a faster FPS (**F**rames **P**er **S**econd) not by optimizing everything but by optimizing parts of the code that do a lot of work especially in sections that get called often.

**Note however** you shouldn't try being a smarta██ and confidently make assumptions.

For instance, one shouldn't confidently assume that by moving the game onto an SSD does the **F**rames **P**er **S**econd increase, that is a confidently wrong assumption backed up by nothing but one's own b███s███. A faster disk read/write speed does not equate to being able to process code faster, the bottlenecks with FPS lie with the CPU, GPU, and RAM.

Trying to presumptuously force such brazen assumption that is already proven wrong by even basic computer hardware classes not only wastes your own and other developers' time but it makes you overall a less trustworthy person to talk technical details with.

It's great if one already has the experience and knowledge to propose a maintainable yet performant design but this isn't always the case, and it's more likely that to get to performant design it's a team effort to incrementally — step by step — reach that goal.

One can incrementally implement the ideal optimized code they have in mind through measuring their code with profilers and doing the necessary research. **To make stuff up without the qualifcations or proof and to directly cite from a Large Language Model / Generative AI hinders this goal and can even reverse progress.**

Don't make stuff up just because you want to *appear* clever for what reason? It'll result in spaghetti code nobody understands and will likely run slower. [KISS principle](https://en.wikipedia.org/wiki/KISS_principle) or whatever.

### Error Handling

It is inevitable that the game will enter an invalid state. These invalid states should avoid making the game panic and crash.

The likelihood of this occuring can be massively reduced through [static typing](https://docs.godotengine.org/en/stable/tutorials/scripting/gdscript/static_typing.html) and [automated testing](https://godot-gdunit-labs.github.io/gdUnit4/latest/).

Otherwise, try to give the user a good time when an irrecoverable error occurs or at least soften the blow.

### Observability

Games are a long running process with a vast amount of parts to it thus important to know goes on inside it.

This can be accomplished using the `print_debug(...)` and `print_stack()` functions. Combined with `push_error(...)` and `push_warning(...)` these functions provide both contributors and the engine alike with a rich amount of information enough to trace where something had happened.

If the game ever enters an invalid state, you may immediately halt the execution of the game by inserting an `assert` statement. The game can not recover from this state and as such you are forced to restart the game.

Alternatively, you may create [breakpoints](https://docs.godotengine.org/en/4.4/tutorials/scripting/debug/debugger_panel.html#stack-trace) either through the script editor or by inserting the `breakpoint` keyword. Breakpoints pauses execution of code at that line and may be continued.

Avoid using the plain old `print(...)` function. It does not tell contributors where in the code base was the `print()` function called.

Avoid naming things `Manager`s. You already manage data and functions when programming thus by naming things `SomethingManager` you are actively harming the time and efforts of other developers with [a name that explains absolutely nothing about what it actually does](https://softwareengineering.stackexchange.com/a/129566).
